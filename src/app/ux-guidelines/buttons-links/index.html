In SKY UX, the uses of hyperlinks and buttons are guided by two interacting principles: style and implementation. The style principle determines the appropriate appearance of the affordance to the user. The implementation principle involves the core HTML – whether the <a> tag or the <button> tag are appropriate for a situation. In SKY UX, these guidelines overlap and interact: differences in styling do not need to correspond to the use of different tags.

Style

SKY UX uses several styles for hyperlinks and buttons. The differences in their appearance carry meaning about their roles in the system, distinguishing between navigation and action, or conveying different levels of emphasis/priority.

Primary actions

The stye of a “primary action” uses bold, saturated color to convey importance and drive user workflows. There should only be one primary action available in any given system state. Use primary actions for:

  • The most important (or frequent) action on a page
  • The “progressive” terminal action on a dialogue (e.g. “Save”)

Secondary actions

Secondary actions are less important or less frequently used than primary actions. Since secondary buttons are less important than primary buttons, SKY UX reduces the visual weight to make them noticeable but not attention-drawing. Use secondary actions for:

  • Toolbar or action bar buttons
  • Actions relevant to only part of a page’s content (cards, tiles, etc.)

Tertiary actions

Tertiary actions have a lighter visual weight, but invisibly occupy the same amount of space as primary and secondary actions. Use tertiary actions for:

  • Infrequently used actions or options on a page
  • The “regressive” action on a dialogue (e.g. “Cancel” or “Close”)

In-line links

In-line links are given the least visual weight and appear in-line with text instead of having their own rectangular space. They have no “button” padding or spacing and should not be used in groups of other buttons. Take care not to “stack” multiple in-line links too closely together, since this creates usability risks of mis-clicks or mis-touches. In-line links should almost always be used for pure navigation:

  • Parent-to-parent
  • Parent-to-child
  • Links to external applications/services (e.g. a street address that links to a map application)

Implementation

While SKY UX styling properties allow <a> and <button> elements to appear interchangable, subtle differences in their behavior make it important to select the correct implementation. Most notably, <a> tags differ from their <button> counterparts in:

  • Right-click options
  • Long-touch / force-touch behaviors (mobile browsers)
  • Keyboard shortcuts
  • Behavior with screen readers and other accessibility software

Determining the correct HTML tag to use for an affordance in SKY UX is done with a simple rule:

  • If the ONLY result of clicking/tapping/selecting an affordance is to load a new URL in the browser, then the affordance should be implemented as an <a> tag
  • Otherwise, implement the affordance as a <button>

In other words, <a> elements are only appropriate when an affordance changes nothing about the state of the system and the only result is that the user traverses from one view of content to another. For example, a record page may contain an affordance for deleting itself (“Delete this record”). If the user activates this affordance, the browser will indeed redirect to a new URL (since the existing page will presumably cease to exist), but since there is also an effect on the system (data is deleted), this affordance should be implemented as a <button>.
