How to use SKY UX to deliver highly accessible experiences

The SKY UX code framework has a lot of accessibility goodness built-in out of the box - good semantic markup, descriptive text equivalents, support for keyboard use, meets color contrast success criteria, and ARIA roles and properties included within components. Individual components also have accessibility guidelines where relevant. If you're using SKY Builder to build your application, it provides an automated accessibility testing tool you can use test for some accessibility criteria.

While the code framework is one ingredient in helping you deliver highly accessible experiences, it doesn't get you all the way there. To ensure users with disabilities are able to use your application(s), here are additional practices, guidelines, and tools to ensure they are accessible:

Is SKY UX accessible?

An accessible UX framework is an important part of Blackbaud’s commitment to deliver solutions that help drive positive change in the world. We endeavor to meet the World Wide Web Consortium’s Web Content Accessibility Guidelines 2.0 Level A and AA success criteria. In SKY UX, we build support for accessibility into our components.

Known issues

Accessibility is a work-in-progress and something we focus on improving within SKY UX. When we identify accessibility issues with SKY UX components, we log the issues in the SKY UX GitHub repo.

Get involved

You can help make SKY UX accessible for everyone.

  • Help fix an existing accessibility issue.

  • Highlight an accessibility issue on the SKY UX Slack channel, or file an issue with an accessibility label on the SKY UX GitHub repository.
