This reference describes the commands and options that are available in the SKY UX CLI.

 Commands

  • skyux new — Initializes a new local SKY UX project by pulling down the latest version of the template and installing the necessary dependencies. For information about the default items that the SKY UX application template provides in a SKY UX single-page application, see the template items reference.

  • skyux serve — Serves the current SKY UX single-page application locally. This command also opens the host endpoint with the necessary query string to pull your local files. It tacks on local=true in the query string to instruct SKY UX Host to pull files from the local machine instead of the hosted location. By default, the SPA automatically opens in the default browser, but you can specify a different browser with the --browser option. Or you can add the --launch none option to prevent the SPA from opening. The serve command basically updates the production location rather than creating a duplicate local instance, so to view the production version alongside a local version, you need to open a separate incognito tab. The serve command sets a cookie to apply local dev mode.

  • skyux build — Builds the current SPA into the dist/ folder. The build and serve commands are largely the same command under the hood, but serve works in memory and launches a web server for the user, while build performs the same operations by writing and storing files to disk. The build command produces the static assets for an SPA.

  • skyux test — Runs unit tests. The Karma unit test is configured under the hood to look for specific file names, similar to how SKY UX Builder looks for index.html files. The Karma unit test looks for any files in the src/app folder with names that end in .spec.ts. Users should write separate tests to confirm that the components work. For more information about unit tests, see the unit test reference.

  • skyux e2e — Runs end-to-end tests that test your SPA running in a real browser, interacting with it as users would. The e2e command uses a tool called Protractor to look for files in the e2e folder with names that end in .e2e-spec.ts. End-to-end tests supplement the Karma unit test, but end-to-end files are written slightly differently. Both test files use a syntax called Jasmine. For more information about end-to-end tests, see the end-to-end test reference.

  • skyux watch — Runs the test command and watches the file system for changes. The watch command monitors changes in much the same way as the serve command, but for testing, the watch command is always running, much like debug mode in Visual Studio. If you change a test, the window is watching that and runs the test. This command is useful when writing a bunch of tests.

  • skyux version or skyux -v — Returns the version of SKY UX CLI.

  • skyux, skyux help, or skyux -h — Displays help information for SKY UX CLI arguments. This lists the commands and options that are available in the SKY UX CLI and provides descriptions.

Options

  • --assets — Specifies where to locate static assets when skyux build runs. The skyux build command replaces any instances of ~/assets/ with this value in HTML, CSS, and SCSS files. For example, skyux build –-assets https://example.com/assets. By default, the command replaces ~/assets/ with the CDN. For information about how to include static assets in HTML, CSS, and SCSS files, see the static assets tutorial.

  • --browser or -b — Specifies a browser to open the application in when skyux serve runs. For example, skyux serve --browser firefox. This option is useful for cross-browser manual testing, and the available values are chrome, edge, firefox, iexplore, and safari. By default, skyux serve opens the SPA in the default browser.

  • --envid — Specifies the environment ID to add to the Host URL when skyux serve runs. For example, skyux serve --envid 1234.

  • --launch or -l — Specifies the URL to launch when skyux serve runs. For example, skyux serve --launch local launches the local URL. The available values are none, which stops any URL from launching, local, and host, which launches the SKY UX Host URL and is the default value. For more information about this option, see the tutorial on how to serve the default single-page application.

  • --no-coverage — Prevents the instrument loader from minifying files to enable debugging in browsers when skyux watch runs. By default, the instrument loader minifies files, which makes the code unreadable. This option prevents minifying so that developers can view the original source in the console while debugging the code.

  • --svcid — Specifies the service ID to add to the Host URL when skyux serve runs.

  • --template or -t — Specifies the base template to use when creating a SKY UX application. By default, skyux new uses the SKY UX application template that includes the necessary items for a SKY UX single-page application.
