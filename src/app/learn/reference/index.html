Technical reference

This technical reference describes the commands and options that are available in the SKY UX CLI.

Commands

  • skyux new — Initializes a new local SKY UX project by pulling down the latest version of the template and installing the necessary dependencies.
  • skyux serve — Serves the current single-page application locally. This command also opens the host endpoint with the necessary query string to pull your local files. It tacks on local=true in the query string to instruct SKY UX Host to pull files from the local machine instead of the hosted location. The SPA automatically opens in the default browser, but you can add the --launch none option to the command to prevent the SPA from opening. The serve command basically updates the production location rather than creating a duplicate local instance, so to view the production version alongside a local version, you need to open a separate incognito tab. The serve command sets a cookie to apply local dev mode.
  • skyux build — Builds the current SPA into dist/. The build and serve commands are largely the same command under the hood, but serve works in memory and launches a web server for the user, while build performs the same operations by writing and storing files to disk. The build command produces the static assets for an SPA.
  • skyux version or skyux -v — Returns the version of the SKY UX CLI that is currently in place.
  • skyux test — Runs unit tests tests. The Karma unit test is configured under the hood to look for specific file names, similar to how SKY UX Builder looks for index.html files. The Karma unit test looks for any files with a name that ends in ".component.spec.ts." Users should write separate tests to confirm that the components work.
  • skyux watch — Runs the test command and watches the file system for changes. The watch command monitors watches for changes in much the same way as the serve command, but for testing, the watch command is always running, much like debug mode in Visual Studio. If you change a test, the window is watching that and runs the test. This command is useful when writing a bunch of tests.
  • skyux e2e — Runs end-to-end tests that test your SPA running in a real browser, interacting with it as users would. This command uses a tool called Protractor to look for files that end in ".e2e-spec.ts." End-to-end tests supplement the Karma unit test, but end-to-end files are written slightly differently. Both test files use a syntax called Jasmine.
  • skyux, skyux help, or skyux -h — Displays a list of the commands and options that are available in the SKY UX CLI and provides descriptions.

Options

  • -l or --launch — Specifies the URL to launch when skyux serve runs. For example, skyux serve --launch local launches the local URL. The available options are none, which stops any URL from launching, local, and host, which launches the host URL and is the default value.
  • --noServe — Stops the serve command from automatically running when calling skyux e2e.
