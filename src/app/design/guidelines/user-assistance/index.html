Persistent inline help

Sometimes a small amount of persistent inline explanatory text can help users understand tasks. For example, persistent inline help is very useful for tasks that are complex and infrequently performed. It is also helpful for tasks where user actions have far-reaching consequences that are not obvious.

However, the scenarios that require persistent inline help text are fairly rare. For example, persistent inline help text is not appropriate for frequently performed tasks because the value diminishes over time even though the text continues to take up screen space.

You should limit persistent inline help to one or two sentences because blocks of persistent text can clutter pages and because users almost always ignore large blocks of text. And to minimize visual clutter, you should use typography to de-emphasize persistent inline text.

Invoked inline help

For scenarios where inline help text should not be persistent, you can give users the ability to invoke it. For example, in scenarios where explanatory text is useful when users first perform tasks but the ramifications of mistakes are small, you can provide a help button to invoke explanatory help text. Depending on the complexity of the help text, you can display it in a popover or the help panel. Invoked inline help text allows users to access to the help text as necessary but avoids most of the visual clutter from persistent inline help.

Page- and modal-level help

When users perform complex tasks, they may require comprehensive documentation for the entire task instead of inline help for individual components of the task. Since this documentation applies to more than a single field, users should invoke it at the page or modal level, depending on the context. In Blackbaud products, users can invoke the help service from the omnibar or a modal header bar.
