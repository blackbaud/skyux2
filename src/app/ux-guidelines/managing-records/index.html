Add records
Add from a list

This is likely to be the most common interaction by which records are added. The user will go to a list of existing records (and maybe search for the specific record) to add the new record. This list could be its own page, or within a tile on a page. The add button will open a modal form.

Add another

There will be cases where a user needs to add several records in sequence. If these records are only a few fields, they can use this pattern along with an inline add form. The most likely scenario for this to come up is while filling out a form that can have multiple instances of a child record added at the same time. For example, while adding a new constituent the user may want to add multiple phone numbers.

Add from a search field

A record may need to reference another record, such as a donation that references a constituent. There will be cases where the referenced record does not exist yet, and will need to be added on the fly. The user should search first to ensure that they do not create a duplicate record. Once they know the referenced record does not exist, its add form can be invoked.

Adding child records

Some record types have a lot of child records associated with them. For example, a constituent record will have contact information, relationships and notes, among other information. While some of this will be known when the parent record is created, much will be added after the fact. Rather than adding all this information through a master form, it should be added from the parent record page. The method used to add the information will depend on the other guidelines here and in the Forms guidelines.

Edit records
Edit from a list

If a user needs to edit a record they should be able to do so from a list of those records via the context menu, without having to go to the specific record page itself. As with adding a record from a list, if the record is complex the user may only be able to edit core information in this way and will have to go to the record page to edit ancillary information. This guideline applies to full page lists and lists within a tile.

Edit from a record page
Core information

If a user is on a record page and needs to edit the core information of the record they should be able to open the edit form from the page summary toolbar.

Non-list information

Some types of information will be presented in a non-list format, such as a definition list or text block. The edit for these should be invoked via an edit button on the top right side of the content block.

Tiles with multiple types of non-list information

A page may include tiles that contain multiple types of information in a compact manner. In these cases the edit can be invoked via a split edit button in the tile toolbar, to avoid having multiple edit buttons in close proximity within the tile.

Remove records
Delete records
From a list

Records that are less systemically important or have a low cost of re-creation in the case of accidental deletion can be deleted via the context menu in a list.

From a record page

Systemically important records should be deleted from the record page, rather than a list. This provides an extra level of safety to prevent accidentally deleting a record incorrectly by requiring the user to take extra steps. Deleting important records should be a rare occurrence so this extra interaction will not be overly burdensome.

Removing temporary items

There will be some types of information that are only needed temporarily, such as list filters. The user should remove these via a button using the “fa-close” icon.

Mark records inactive

Some types of information need to be preserved in the system beyond their active usefulness for audit or historical reference purposes. Since this action is reversible it can be accessed from lists even for systemically important types of records.
